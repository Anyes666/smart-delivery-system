## 二、侧边栏（配置面板）

侧边栏是系统的控制中枢，从上到下依次为：

### 2.1 数据源选择

**位置**：侧边栏顶部，「📂 数据源」标题下方

**两种模式**，通过单选按钮切换：

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **使用现有文件** | 下拉框列出 `data/` 目录下所有 CSV 文件 | 日常使用，选已有数据 |
| **上传 CSV** | 拖拽或点击上传新的 CSV 文件 | 评委现场换数据、临时测试新城市 |

**CSV 格式要求**：

```csv
id,x,y,demand,time_window_start,time_window_end,priority
0,116.397,39.908,0,0,86400,0
1,116.342,39.951,5,28800,43200,2
```

| 列名 | 含义 | 示例 | 必填 |
|------|------|------|:--:|
| `id` | 点编号。**0 必须为仓库** | 0, 1, 2... | 是 |
| `x` | 经度（WGS-84 坐标系） | 116.397 | 是 |
| `y` | 纬度（WGS-84 坐标系） | 39.908 | 是 |
| `demand` | 需求量。仓库填 0 | 5 | 是 |
| `time_window_start` | 时间窗开始（秒，0-86400） | 28800 (=8:00) | 否，默认 0 |
| `time_window_end` | 时间窗结束（秒） | 43200 (=12:00) | 否，默认 86400 |
| `priority` | 优先级（0-10，越大越紧急） | 2 | 否，默认 0 |

**上传后的处理**：文件自动保存到系统临时目录，然后被 `data_processing.py` 解析。上传的 CSV 会触发与现有文件相同的验证流程——缺少必要列会直接报错。

**切换数据源的影响**：切换文件后，侧边栏下方的数据预加载区会自动重新解析新文件，更新城市检测结果。之前优化运行的结果会被清空（需要重新点击「开始优化」）。

---

### 2.2 优化参数

**位置**：数据源下方，「🎛️ 优化参数」标题下方

#### 最大车辆数（滑块）

| 属性 | 值 |
|------|-----|
| 类型 | 整数滑块 |
| 范围 | 1 ~ 30 |
| 默认值 | 5 |
| 步长 | 1 |

**含义**：系统可调用的最大车辆数量。注意这不是实际使用的数量——OR-Tools 求解器会根据「固定成本 vs 可变成本」自动决定实际启用几辆。设置偏大不会浪费，设置偏小可能导致溢出（部分订单无车可送）。

**实际案例**：设置 20 辆车，但 17 个客户的配送可能只需要 3 辆。求解器通过 `SetFixedCostOfVehicle` 机制自动决策——多派一辆的成本 > 现有车多跑一段的成本时，就不派新车。

#### 求解算法（下拉框）

| 算法 | 内部实现 | 适用场景 |
|------|---------|---------|
| **OR-Tools (推荐)** | Google OR-Tools 路径优化求解器，GUIDED_LOCAL_SEARCH 策略 | 追求最优解，20 点以内效果极佳 |
| **贪心算法** | 最近邻启发式 + 车型匹配 | 快速出结果，适合大规模（50+ 点）或演示 |
| **聚类优先** | KMeans 聚类 + 簇内最近邻 TSP | 配送点自然分组时效果好（如多个城区）|

**推荐**：比赛演示始终选 OR-Tools。

#### 使用当前真实时间（复选框）

| 属性 | 值 |
|------|-----|
| 默认 | ✅ 勾选 |

**勾选时**：系统用你电脑当前的时钟时间作为模拟时间。高德 API 返回此刻真实路况的行程时间，数据采集器标记为 `real_time=True`。这是积累高质量训练数据的正确方式。

**不勾选时**：系统用 `settings.py` 中配置的默认模拟时间（10:00 AM）。适合快速测试、反复对比同一时刻的路线效果。

### 2.3 数据预加载区

**位置**：参数配置与运行按钮之间

当选择了数据源后，系统会自动调用 `get_data_summary()` 解析文件。如果 CSV 格式有误，这里会显示红色错误提示。正确解析后会显示通过（无额外 UI 元素）。这是静默的后台操作，目的是让 Tab 1 数据概览始终有内容可显示。

---

### 2.4 开始优化按钮

| 属性 | 值 |
|------|-----|
| 样式 | 蓝色全宽按钮（`type="primary"`）|
| 状态 | 数据文件未选择时灰色禁用 |

**点击后发生的事情**（完整数据流）：

```
1. st.spinner 显示加载动画 + "正在调用高德 API..."
2. 创建 DeliverySystem 实例 → 初始化高德地图网络
3. 加载 CSV → 解析配送点 → 计算距离矩阵（调用高德方向 API）
4. 加载 LightGBM 模型 → 加载在线自适应预测器
5. _apply_traffic_penalties() → 逐边预测拥堵乘数
6. OR-Tools 求解器 → 异构车队 VRP → 输出路线
7. 生成 Folium 数字孪生地图 HTML
8. 路线结果写入 st.session_state
9. 显示绿色成功提示 + 耗时
```

**耗时预估**：
- 18 个配送点：约 30-50 秒（主要耗时在高德 API 批量查询距离矩阵）
- 50 个配送点：约 2-4 分钟
- 100 个配送点：约 5-10 分钟

**失败处理**：高德 API 超时或网络故障会在 3 次重试后回退到欧氏距离（并在日志中记录警告）。求解器超时（默认 30 秒）会自动放宽时间窗约束重试。

## 三、主界面 — Tab 1：数据概览

**进入条件**：侧边栏已选择有效数据文件

### 3.1 五项指标卡

页面顶部 5 个等宽卡片，蓝底白字，数据来自 `get_data_summary()`：

| 指标 | 数据来源 | 含义 |
|------|---------|------|
| **配送点总数** | `len(coords)` | CSV 中所有点（含仓库）|
| **客户数** | `len(coords) - 1` | 需要配送的客户数量 |
| **总需求量** | `sum(demands)` | 所有客户需求量之和，单位与 CSV 一致 |
| **有时间窗** | 时间窗 ≠ `(0, 86400)` 的数量 | 有时效要求的订单数 |
| **覆盖面积** | `Δlon × 111.32 × Δlat × 111.32` | 配送区域近似面积（km²）|

**面积计算说明**：使用近似公式 `1°经度 ≈ 111.32km × cos(lat)`。实际上对不同纬度会有误差，但对北京（~39.9°N）来说，1°经度 ≈ 85.4km。这里的 111.32 是赤道值，会略偏高。作为估算值足够用。

---

### 3.2 地理信息行

指标卡下方 3 列并排显示：

| 字段 | 数据来源 | 含义 |
|------|---------|------|
| **检测城市** | `detect_city_from_points(coords)` | 基于 0.5°×0.5° 网格识别，支持北上广深等 8 个城市 |
| **经度范围** | `min(lons) ~ max(lons)` | 最小/最大经度 |
| **纬度范围** | `min(lats) ~ max(lats)` | 最小/最大纬度 |
| **高优先级** | `sum(p > 0 for p in priorities)` | 优先级大于 0 的订单数 |

**城市检测逻辑**（来自 `travel_time_predictor.py`）：计算所有配送点的几何中心，然后与预设的 8 个城市范围（北京/上海/广州/深圳/成都/武汉/杭州/南京）进行经纬度区间匹配。不匹配任何已知城市时，返回 `city_{lon}_{lat}` 格式的哈希标识。

**为什么需要城市检测**：不同城市的 LightGBM 模型和自适应预测模型是独立存储的（文件名为 `travel_time_lgb_{city}.pkl` 和 `congestion_{city}.pkl`）。正确检测城市才能加载对应的模型。如果检测到的城市没有模型，系统自动回退到定速公式。

## 四、主界面 — Tab 2：路线结果

**进入条件**：点击「开始优化」后优化成功完成

### 4.1 五项结果指标卡

| 指标 | 含义 |
|------|------|
| **总行驶距离** | 所有车辆行驶距离之和（km），来自高德真实路网 |
| **总成本** | 统一成本度量（km-当量）= 固定成本 + 运行成本。1 km-当量 ≈ 小型车行驶 1km 的成本 |
| **使用车辆** | 实际被启用的车辆数。可能远小于「最大车辆数」|
| **是否溢出** | 有未配送的客户时显示「是」并注明未送单数 |
| **路线数** | 等于使用车辆数（每车一条路线）|

### 4.2 车辆路线详情卡片

每条被激活的路线占一个灰色卡片，左侧蓝色边框。

**卡片内容**：

```
🔵 车辆 1 — 小型车 — 3 站 | 15.1 km | 载重 30/30 (100%) | 固定成本 5 km-eq | 运行成本 12.1 km-eq
```

| 字段 | 含义 | 示例解读 |
|------|------|---------|
| **图标** | 🔵小型 🟡中型 🔴大型 | 车型可视化区分 |
| **车辆编号** | route_id + 1 | 从 1 开始，对应仓库出发的路线 |
| **车型** | 小型车/中型车/大型车 | 求解器根据订单需求自动匹配 |
| **站数** | 该车停靠的客户数（不含仓库） | "3 站" = 送 3 个客户 |
| **距离** | 该车路线总长（km） | 15.1 km |
| **载重** | 实际装载量 / 容量上限 | 30/30 = 满载 |
| **载重率** | load/capacity × 100% | 100% = 满载，50% = 半空 |
| **固定成本** | 车辆启用成本（km-当量） | 小型车 5，中型车 10，大型车 18 |
| **运行成本** | 行驶距离 × 该车型的单位成本 | 小型车 ×0.8，中型车 ×1.0，大型车 ×1.2 |

**成本模型解读**：
- 如果小型车送完所有 17 个客户但满载率 100%，总成本 = 5 + 12 = 17 km-eq
- 如果改用大型车（固定成本 18），虽然一车搞定但成本可能更高
- OR-Tools 正是在这个权衡中寻找最优：多派一辆小车 vs 一车跑很远

## 五、主界面 — Tab 3：交互地图

**进入条件**：优化成功完成，`folium_map_*.html` 文件生成

**技术实现**：`st.components.v1.html()` 直接嵌入 Folium 生成的完整 HTML 文件。高度固定为 600px，超出部分可滚动。

**地图内容**（由 `EnhancedMapRenderer` 生成）：

| 元素 | 说明 |
|------|------|
| **底图** | 高德瓦片（或 CartoDB，取决于 `TILE_PROVIDER` 配置）|
| **仓库标记** | 绿色房子图标，标注「配送中心」|
| **客户标记** | 蓝色圆点，标注客户编号 |
| **路线** | 彩色折线（不同车辆不同颜色），沿高德折线贴路显示 |
| **Polyline** | 来自高德方向 API 返回的实际道路坐标，非直线 |

**交互功能**：
- 鼠标滚轮缩放
- 拖拽平移
- 点击标记查看详情
- 右上角图层切换（如果 Folium 配置了多个 TileLayer）

**注意事项**：
- 地图较大（通常 200KB-500KB），首次加载需要 1-3 秒
- 如果地图不显示，检查 `output/folium_map_*.html` 是否存在
- 高德瓦片在国内访问极快，海外可能较慢

## 六、主界面 — Tab 4：AI 模型

**进入条件**：任意时刻都可查看。点击「刷新模型状态」按钮获取最新数据。

三列布局（LightGBM | 自适应预测 | 训练数据）：

### 6.1 LightGBM 行程时间预测模型

| 显示内容 | 数据来源 | 含义 |
|---------|---------|------|
| **训练状态** | `ml.is_trained` | 绿色=已训练 / 黄色=未训练 / 红色=加载失败 |
| **样本数** | `metrics.n_samples` | 用于训练该模型的样本总数 |
| **RMSE** | `metrics.rmse_seconds` | 均方根误差（秒）。越低越好 |
| **MAE** | `metrics.mae_seconds` | 平均绝对误差（秒）。越低越好 |
| **vs 定速提升** | `metrics.improvement_pct` | 相比 40km/h 定速公式，RMSE 降低了百分之多少 |

**未训练状态的触发条件**：该城市模型文件（`travel_time_lgb_{city}.pkl`）不存在，或样本不足 500 条。此时系统在优化中自动回退到 `dist_km / 40km/h × 3600` 定速公式。

**训练方法**：
```bash
python scripts/train_travel_time.py --train --city Beijing
```

### 6.2 在线自适应行程时间预测器

| 显示内容 | 数据来源 | 含义 |
|---------|---------|------|
| **训练状态** | `adaptive.is_trained` | 绿色=在线学习中 / 黄色=未训练 |
| **更新次数** | `metrics.updates` | 模型参数被 SGD 更新的总次数 |
| **Running Loss** | `metrics.running_loss` | 指数平滑后的 MSE Loss。越接近 0 越准 |

**定位**: 在线自适应行程时间预测 (Online Adaptive Travel Time Prediction)。本质是一个神经网络回归模型 (10维→64→32→1)，每次高德 API 返回真实耗时后通过 SGD 在线更新权重，配合逐渐衰减的探索噪声实现持续自适应，越跑越准。

**Loss 解读**：
- `0.00 ~ 0.05`：极好（预测与真实几乎一致）
- `0.05 ~ 0.20`：良好（日常使用级别）
- `0.20 ~ 0.50`：中等（数据量不足或时段单一）
- `> 0.50`：需要更多数据

### 6.3 训练数据

| 显示内容 | 含义 |
|---------|------|
| **总样本** | 所有城市的 `travel_time_samples_*.jsonl` 总行数 |
| **{city} 样本** | 当前城市的数据量 |
| **还需 X 条** | 距 500 条训练门槛还差多少。≥500 时显示绿色「可训练」|

**数据采集建议**（页面底部提示）：
```bash
python scripts/collect_all_day.py -d my_points1.csv --hours 8,10,12,14,16,18,20,22
```

## 七、Session State（跨页面状态保持）

Streamlit 默认每次交互都会重新执行整个脚本。以下变量通过 `st.session_state` 持久化：

| 变量名 | 类型 | 默认值 | 何时更新 |
|--------|------|--------|---------|
| `optimization_done` | bool | False | 优化成功时 → True |
| `result` | dict / None | None | 优化成功时填充 |
| `city` | str | "Beijing" | 数据加载时自动检测 |
| `map_html` | str / None | None | 优化成功时读取 HTML 文件 |
| `model_status` | dict / None | None | 运行优化后或点击刷新时 |
| `data_summary` | dict / None | None | 侧边栏选择文件后自动填充 |
| `routes_detail` | list / None | None | 优化成功时填充 |

**为什么要用 Session State**：防止 Streamlit 重新渲染时清空优化结果、地图和模型状态。用户在不同 Tab 之间切换时，所有数据保持不变。

## 九、故障排查

| 问题 | 原因 | 解决 |
|------|------|------|
| 侧边栏选择文件后无反应 | CSV 格式不符合要求 | 检查列名是否为 `id,x,y,demand,...` |
| 点击开始优化后长时间无响应 | 高德 API 调用缓慢或网络问题 | 等待 1-2 分钟，查看终端日志 |
| 地图 Tab 无内容 | HTML 文件未生成 | 检查 `output/folium_map_*.html` 是否存在 |
| LightGBM 显示红色加载失败 | `.pkl` 文件损坏 | 删除 `.cache/ml/travel_time_lgb_{city}.pkl` 重新训练 |
| 自适应预测显示红色加载失败 | `.pkl` 文件损坏 | 删除 `.cache/adaptive/congestion_{city}.pkl` |
| 上传 CSV 后报错 | 文件编码不是 UTF-8 | 用记事本另存为 UTF-8 编码 |
| 地图中路线显示为直线 | 高德折线解析失败 | 检查 AMAP_CONFIG 中 api_key 是否正确 |
