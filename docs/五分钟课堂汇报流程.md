## 环节一：开场（30秒）

> **口头说**：
> "城市配送有三个痛点——地图算的是直线距离、忽略了堵车时间、车辆类型一刀切。我们的系统用三层优化逐一解决。"

---

## 环节二：架构（60秒）

**PPT 放这张图**：

```
配送点 CSV ──→ 高德地图 API (真实路网)
                  │
         ┌────────┼────────┐
         ▼                 ▼
      LightGBM        在线自适应预测
    (行程时间预测)    (神经网络+SGD)
         │                 │
         └────────┬────────┘
                  ▼
         OR-Tools 求解器
         (4种国标车型自动选型)
                  │
                  ▼
      交互式数字孪生地图 + 路线动画
```

> **口头说（每层一句话）**：
> - L1 距离：高德 API 获取真实路网距离，不是欧氏直线
> - L2 时间：LightGBM 15维特征预测行程时间，跨城市可迁移
> - L3 拥堵：在线自适应预测器，每次真实数据反馈后自动 SGD 更新，越跑越准
> - 求解层：OR-Tools + 异构车队（微型/轻型/中型/重型货车）自动匹配最优组合

---

## 环节三：界面演示（120秒）— 核心

### 步骤 1：侧边栏配置（15秒）

1. **数据源** → 下拉选 `shanghai_20_houses.csv`（上海 20 个住宅区，展示效果好）
2. **车辆数量** → 拖到 5
3. **优化算法** → 保持 `ortools`
4. 点击 **「开始优化」**

> "这里选了上海 20 个配送点的真实数据，5 辆车，用 OR-Tools 求解。"

### 步骤 2：数据概览 Tab（15秒）

点击 **「📊 数据概览」** 标签。

> "系统自动检测到这是上海的数据，展示了 20 个配送点、总需求量、时间窗分布。覆盖范围是系统用 GPS 坐标自动计算的。"

### 步骤 3：路线结果 Tab（30秒）

点击 **「🚚 路线结果」** 标签，等优化完成。

> "这才是核心——四辆车被派出，总距离 XX 公里，总成本 XX。每辆车都自动匹配了最合适的车型。下面有车辆路线的详细表，每站的时间、距离一目了然。"

### 步骤 4：交互地图 Tab（40秒）

点击 **「🗺️ 交互地图」** 标签。

> "这是数字孪生地图。每种颜色是一辆车的路线——注意看，路线是**贴合真实道路**的，因为底层加载了高德地图瓦片，不是那种直来直去的图。
> 左边有个图层控制器，可以切换拥堵热力图、红绿灯标记。"

操作演示：
- 鼠标滚轮缩放地图
- 点击某条路线看详情弹窗
- 勾选/取消图层控制项

### 步骤 5：AI 模型 Tab（20秒）

点击 **「🤖 AI 模型」** 标签。

> "这个是模型状态面板。左边 LightGBM 行程时间预测，中间是在线自适应预测器——它会随着每次 API 返回自动更新权重，loss 越接近 0 越准。右边是各城市的训练数据量统计。"

## 环节四：总结（30秒）

> **口头说**：
> "三个核心亮点：
> 1. **真实路网替代直线** — 高德 API，距离算得准
> 2. **AI 预测替代定速** — LightGBM + 在线自适应，时间算得准
> 3. **异构车队自动选型** — 4 种国标车型，成本控得准
>
> 适用场景：物流配送、外卖调度、应急物资运输。谢谢！"

## 故障应对

| 情况 | 处理 |
|------|------|
| 忘记预热缓存 | 演示时老实说"正在调用高德 API，大约需要 1 分钟"，边聊架构边等 |
| `启动界面.bat` 打不开 | 终端手动 `streamlit run app.py` |
| 高德 API 超时 | 说"教室网络限制"，展示已有的 output/ HTML 文件 |
| 浏览器没弹出 | 手动打开 `http://localhost:8501` |
| 界面中文乱码 | 按 `Ctrl+0` 重置缩放，或换 Chrome |
